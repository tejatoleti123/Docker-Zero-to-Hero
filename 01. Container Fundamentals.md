## ğŸ“¦ What Is a Container?

A container is a lightweight, portable, and isolated environment that packages:

- Application code  
- Runtime  
- System libraries  
- Dependencies  
- Configuration  

All bundled together so the application runs consistently across different environments.

In simple terms:

A container ensures that an application behaves the same way everywhere.

---

## ğŸ–¥ The Problem Before Containers

Before containers, applications were typically deployed in one of these ways:

1. Directly on a physical server  
2. Inside virtual machines (VMs)

Each method had problems.

### Physical Server Deployment

- Dependency conflicts  
- Hard to isolate applications  
- Difficult to scale  
- Environment inconsistencies  

### Virtual Machines

Virtual machines improved isolation, but introduced:

- Heavy resource usage  
- Full guest operating system per VM  
- Slower startup times  
- Larger storage requirements  

Each VM contains:

- Guest OS  
- Application  
- Libraries  
- Hypervisor layer  

This makes VMs powerful but heavy.

## ğŸ§  Why Containers Are Lightweight

Containers do not package a full operating system.

Instead, they share the host kernel and isolate applications using Linux features:

- Namespaces  
- Control Groups (cgroups)  

We will briefly explain these next.

---

## ğŸ”’ Namespaces (Isolation)

Namespaces provide isolation.

They allow each container to have its own:

- Process tree  
- Network stack  
- File system view  
- Hostname  
- User IDs  

This means a process inside one container cannot see processes in another container.

It feels like its own mini system.

---

## ğŸ“Š cgroups (Resource Control)

Control Groups (cgroups) manage resource limits.

They allow you to restrict:

- CPU usage  
- Memory usage  
- Disk I/O  
- Network bandwidth  

For example, you can limit a container to:

- 512MB memory  
- 1 CPU core  

This prevents one application from consuming all system resources.

A container image is a read-only template used to create containers.

It contains:

- Application code  
- Runtime  
- Dependencies  
- Base OS layer  

An image is not running.

It becomes a container only when executed.

Container images are built in layers.

## ğŸ“œ Open Container Initiative (OCI)

Containers follow open standards defined by the Open Container Initiative (OCI).

OCI defines:

- Image specification  
- Runtime specification  

This means containers are not limited to one tool.

Docker is one implementation.

Other container tools follow the same standards.
